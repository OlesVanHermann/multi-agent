# V3 — Boucle verify, autonomie, banc

> **Principe : la complétion se prouve, ne se déclare pas.**
> Sur une tâche à `verify_cmd`, le `DONE`/`SCORE` émis par l'agent devient
> consultatif ; seul le verdict du vérificateur (exécuté par le bridge,
> hors de portée de l'agent) fait foi.

V3-core = trois lots : **C1** (boucle verify au bridge), **C2** (autonomie :
WAL, budgets, escalade, détection de stall), **C0** (banc de mesure).
Plan de référence : `PLAN-DEV-MULTI-AGENT-V3-R2.md` (hors repo).

---

## Rétrocompatibilité (invariant)

Une tâche **sans** `verify_cmd` traverse le bridge à l'identique de v2 :
aucun appel au vérificateur, aucun événement `verify_*` dans le WAL,
réponse publiée telle quelle. L'opt-in est **par tâche**, pas par agent.
Verrouillé par test (`tests/test_verifier.py::TestVerifyGate`).

## C1 — Boucle verify au bridge

### Flux

```
tâche (verify_cmd) → agent travaille → réponse candidate
    │
    ▼
verifier.py (exécuté par le BRIDGE, cwd = projet)
    ├─ règles anti-hacking sur le diff        → violation : escalade directe
    ├─ exécution verify_cmd (timeout, capture)
    │
    ├─ VERT  → git checkpoint + audit origin=verify
    │          réponse publiée avec marqueur [VERIFY_GREEN]
    │
    └─ ROUGE → budget restant ?
         ├─ oui : rapport ré-injecté dans l'inbox (FROM:verify|FAIL|…),
         │        retry++ — l'agent lit l'erreur et répare
         └─ non : [VERIFY_FAILED] BLOCKED|task=…|raison=budget_retries
                  → l'orchestrator lève RuntimeError → on_failure du workflow
```

### Porter une tâche à verify

Workflow YAML (`workflow_engine.py`) :

```yaml
steps:
  - name: ma-tache            # devient task_id (WAL, checkpoints)
    agent: 300
    prompt: "Implémente X."
    verify: "pytest -q tests/test_x.py"   # optionnel — clé V3
```

Orchestrator direct :

```python
send_and_wait(300, "Implémente X.", verify_cmd="pytest -q", task_id="ma-tache")
```

Message Redis brut : champs `verify_cmd`, `task_id` (+ `deadline` epoch
optionnel) sur l'inbox `{MA_PREFIX}:agent:{id}:inbox`.

### Règles anti-hacking (déterministes, `verifier.py`)

Contrôlées sur `git diff HEAD` + fichiers non suivis, AVANT le verify :

1. écriture dans un chemin interdit : `pool-requests/tests/`,
   `bench/oracle/`, `spec/acceptance/` ;
2. plus de 3 `assert` supprimés et aucun ajouté ;
3. marqueurs `skip`/`xfail` ajoutés dans des tests.

Violation → pas de retry, escalade immédiate (`raison=hacking`).

### Protection de l'oracle — double barrière

1. `permissions.deny` des 8 profils `login/claude*/settings.json` :
   `Read/Write/Edit(./bench/oracle/**)`, `Write/Edit(./pool-requests/tests/**)` ;
2. règles anti-hacking ci-dessus (même si la barrière 1 saute).

Verrouillé par `tests/test_no_heldout_leak.py`.

**Migration** : depuis v3.0.1, `patch/upgrade.sh` fusionne automatiquement
les règles deny dans les profils `login/claude*/settings.json` existants
(`patch/merge-deny-rules.py` : union des règles, le reste du fichier —
model, theme… — ne bouge pas). Machine encore en v2 : lancer l'upgrade
**deux fois** — la 1re passe (ancien script) installe le nouvel outillage
`patch/`, la 2de applique les migrations bench/login/prompts. Report manuel
toujours possible : bloc exact dans `login/claude1a/settings.json` du repo.

### Contrat côté agents

`prompts/RULES.md` §10 + section dédiée du template 3XX : une tâche à
verify n'est pas finie tant que le verify n'est pas vert ; lire le rapport
`FROM:verify|FAIL`, réparer, ne jamais toucher aux tests d'acceptation.

## C2 — Autonomie

### WAL (`scripts/agent-bridge/wal.py`)

Stream Redis append-only `{MA_PREFIX}:wal` — **audit + diagnostic**, le
`state_file` du workflow_engine reste la source de vérité de reprise
(le consumer group redélivre déjà les tâches non-ack, A4).

| Événement | Émis quand |
|-----------|------------|
| `task_assigned` | tâche extraite de la queue du bridge |
| `verify_green` | verify vert |
| `verify_red` | verify rouge (champ `retry`) |
| `verify_retry` | rapport ré-injecté, tâche requeue |
| `verify_escalation` | budget épuisé / hacking / deadline (champ `motif`) |
| `api_error_retry` | retry après erreur API |
| `nudge` | relance watchdog (stall) |
| `escalation` | escalade watchdog (`motif=stall`) |

Toutes les émissions sont **best-effort** : une panne Redis sur le WAL ne
tue jamais le thread du bridge. Au démarrage, le bridge journalise la
dernière tâche ouverte (`wal.open_task`) pour le diagnostic post-crash.

### Budgets

- **Retries verify** : `VERIFY_MAX_RETRIES` (défaut 3) par tâche —
  au-delà, escalade `budget_retries`.
- **Temps mural** : champ `deadline` (epoch) sur la tâche — dépassée à la
  prise en charge, la tâche n'est PAS lancée (aucun token brûlé) :
  `[VERIFY_FAILED] BLOCKED|task=…|raison=deadline`. Deadline absente ou
  invalide = comportement v2. La deadline suit la tâche à travers les
  requeues (retry verify, erreur API, compaction).

### Détection de stall (`healthcheck.py`)

Machine à états par agent, dans le watchdog existant :

```
busy + WAL silencieux > WATCHDOG_STALL_THRESHOLD (600 s)
    → alerte warning + nudge dans l'inbox
      ("FROM:watchdog|Où en es-tu ? …" — le nudge est un événement WAL :
       il réarme le compteur, l'agent a une fenêtre complète pour répondre)
toujours silencieux après une fenêtre complète
    → alerte critique + WAL escalation motif=stall, puis silence (pas de spam)
toute activité réelle (événement non-nudge) → réarmement
agent sans aucun événement WAL (bridge v2) → jamais de faux positif
```

### Escalade structurée

Format unique : `BLOCKED|task|raison|question`. L'humain n'est sollicité
que sur ambiguïté réelle, avec une question précise (RULES.md §10).

## C0 — Banc (`bench/`)

```
bench/
├── tasks/{niveau}/{task_id}/task.md   # énoncé visible agent
├── oracle/{task_id}/verify.sh         # tests cachés (deny + anti-hacking)
├── heldout.txt                        # split held-out (~1/3)
├── run.sh / list_tasks.sh             # exécution
├── collect.py / aggregate.py          # métriques → results/*.jsonl
└── tasks/wf-single.yaml               # workflow 1 tâche ({{task}}, {{verify}})
```

### Usage

```bash
# Baseline v2 (dispatch SANS verify — succès mesuré post-hoc par l'oracle)
bench/run.sh v2.12-baseline 5 heldout v2

# v3 (dispatch avec verify=oracle — boucle C1 active)
bench/run.sh v3-core 5 heldout v3

# Deltas appariés par tâche + IC bootstrap 95 %
python3 bench/aggregate.py v3-core v2.12-baseline
```

Métriques par run (`collect.py`) : `success` (oracle post-hoc, verdict
uniforme v2/v3), `done_declared` (l'auto-déclaration, à confronter au
verdict), `cycles_to_green` (`verify_red` + 1), `retries`,
`interventions` (escalades), `hacking_detected`, `wall_s`.

### Règles du banc

- **État de départ figé** : tag git `bench-base-<task_id>` dans `project/`
  (posé à l'import des tâches) — `run.sh` fait un `checkout -f` dessus.
- **Held-out** : jamais lu par les agents, jamais cité dans
  `prompts/`/`templates/` — verrouillé par `tests/test_no_heldout_leak.py`.
- **Import depuis l'historique réel** (`pool-requests/archive/` + `done/`)
  : à écrire sur la machine projet (le format d'archive est propre au
  site) — les 3 tâches synthétiques `s01`–`s03` couvrent les 3 niveaux en
  attendant.
- `bench/results/` est local (gitignoré) ; seul le rapport final se commit.

## Variables d'environnement V3

| Variable | Défaut | Rôle |
|----------|--------|------|
| `VERIFY_MAX_RETRIES` | `3` | Budget de retries verify par tâche |
| `VERIFY_TIMEOUT` | `600` | Timeout (s) d'un `verify_cmd` |
| `PROJECT_DIR` | `$BASE/project` | cwd du verify + des règles anti-hacking |
| `WAL_MAXLEN` | `100000` | Borne du stream `{MA_PREFIX}:wal` |
| `WATCHDOG_STALL_THRESHOLD` | `600` | Silence WAL (s) avant nudge |
| `BENCH_ORACLE_TIMEOUT` | `600` | Timeout (s) de l'oracle post-hoc |
| `BENCH_BOOTSTRAP` | `10000` | Rééchantillonnages bootstrap (IC 95 %) |

## Tests

```bash
python3 -m pytest tests/test_verifier.py tests/test_wal.py \
    tests/test_stall_detection.py tests/test_no_heldout_leak.py -v
```

- `test_verifier.py` — règles anti-hacking, run_cmd, gate complet dans le
  vrai `_process_queue` (rétrocompat v2, vert, rouge→vert, budget,
  hacking, deadline, émissions WAL).
- `test_wal.py` — emit/last_event/open_task sur redis éphémère.
- `test_stall_detection.py` — machine à états nudge→escalade.
- `test_no_heldout_leak.py` — double barrière oracle + anti-fuite held-out.
- Transport : `test_orchestrator.py`, `test_workflow_engine.py`,
  `test_done_signal.py` (origin).
